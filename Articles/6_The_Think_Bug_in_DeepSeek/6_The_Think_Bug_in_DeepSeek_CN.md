### 1. Bug 介绍与复现

#### 1.1 Bug 介绍

大约三天前（2026年5月16日），多位用户报告在使用 DeepSeek LLM 模型网页聊天时出现异常行为。在测试过程中，用户发现如果提示仅包含特殊标记 `<think>` 或不完整标记 `<think`，模型可能会意外地生成不相关的响应、隐藏的推理痕迹，或回答当前用户从未问过的问题。

在一些情况下，生成的内容看起来逻辑结构清晰且上下文连贯。由于其中一些响应看起来像是有效用户查询的片段，因此在线讨论迅速引发了对可能存在的跨会话数据泄露或推理缓存暴露的担忧。

发现时，社区讨论了几个可能的解释：

- **GPU 缓存或负载均衡器缓存泄露**：一些用户怀疑推理服务器缓存未能在请求之间正确清除，可能导致其他用户的提示或推理痕迹片段出现在不相关的会话中。
- **分词器或提示解析器 bug**：其他人认为问题与分词器或内部提示格式化机制有关，其中 `<think>` 标记可能意外触发隐藏的推理模式行为或特殊的内部指令。
- **特殊标记解析问题**：另一种假设是，`<think>` 可能被视为模型在思维链生成过程中内部使用的保留控制标记，当直接暴露给用户输入时会导致输出不稳定。

#### 1.2 Bug 复现

该 bug 相对容易复现。在本实验中，使用了官方的 DeepSeek 网页界面，配置如下：

- **在线搜索功能：** 禁用
- **深度思考功能：** 启用

测试方法很简单：输入提示仅包含文本 `<think>`，并记录返回结果。在独立的聊天会话中进行了七轮独立测试，没有任何对话历史或上下文提示。

现在我们输入单词 `<think>` 并检查答案，我们测试了几轮（七次独立的、没有任何上下文的聊天）

#### 1.3 官方解释与初步讨论

今天（2026年5月19日），官方 DeepSeek 账号发布了解释（“关于 `<think>` 标记导致模型出现幻觉问题的说明”）。根据官方解释，该问题是由一个特殊标记触发了模型内部的异常幻觉行为，并且公司表示没有证据表明存在私人信息泄露。（如下图所示）

![](img/s_09.png)

尽管官方解释将问题归因于特殊标记触发的幻觉，但一个重要的问题仍然存在：

如果异常行为纯粹是由分词器或模型级别的幻觉引起的，那么在完全隔离的本地部署的 DeepSeek 模型上运行，在没有共享推理基础设施的独立 GPU 服务器上，是否也会出现同样的问题？

为了进一步调查这种可能性，我们使用本地部署的 DeepSeek 模型进行了额外的验证实验。

### 2. 验证实验设置与结果

根据 DeepSeek 的官方解释，异常行为是由特殊标记触发的幻觉引起的。如果这个解释是正确的，那么我猜在完全隔离的本地部署的 DeepSeek 模型上运行，在没有共享云推理基础设施或请求缓存系统的独立 GPU 服务器上，也应该能够复现相同的行为。

#### 2.1 实验平台

本次测试使用的实验平台是一台 NVIDIA GB10 DGX-Spark AI 工作站，拥有 128 GB 共享内存。硬件资源足以通过 Ollama 在本地部署和运行更大的 DeepSeek-R1 32B 模型。

实验环境包括：

- **硬件平台：** NVIDIA GB10 DGX-Spark
- **LLM 框架：** Ollama
- **使用的模型：** `deepseek-r1:32b`
- **执行模式：** 完全本地推理，无外部 API 访问

实验的主要目标是确定单独的 `<think>` 标记是否能够一致地触发在官方 DeepSeek 网页服务中观察到的相同异常幻觉行为。

#### 2.2 实验程序

创建了一个简单的 Python 测试程序，用于反复向本地部署的模型发送 `<think>` 标记，并记录推理痕迹和最终响应。

实验代码如下：

```python
from ollama import chat
for i in range(10):
    print("Test Around %s" %str(i))
    response = chat(
        model='deepseek-r1:32b',
        messages=[{'role': 'user', 'content': '<think>'}],
        think=True,
    )
    print('Thinking:\n', response.message.thinking)
    print("Response from Model:")
    print(response.message.content)
```

#### 2.4 初步观察与讨论

基于前 30 个测试用例，本地部署的 DeepSeek-R1 32B 模型未能复现官方 DeepSeek 网页服务中观察到的异常行为。

此后还进行了额外的实验轮次（另外 30 次测试），但在当前实验环境中仍未能复现相同的问题。

### 3. 推测与讨论

基于实验结果和在线讨论，`<think>` 标记问题的确切原因仍未 100% 确定。尽管 DeepSeek 官方解释说异常行为是由特殊标记触发的幻觉引起的，但不同用户的验证结果似乎不一致。

一些人提到，即使在本地部署的环境中，他们也能复现类似的异常输出，而其他测试——包括本文描述的实验——在独立 GPU 执行下未能复现该问题。这表明问题可能不是由单一因素引起的。

一个重要的讨论点是幻觉内容的性质本身。如果生成的“随机”答案包含比模型原始训练截止日期或发布时间更新的信息、事件或引用（正如我们已经发现的），那么问题就变得更加复杂。在这种情况下，就会引发关于响应是模型纯粹产生的幻觉，还是推理基础设施中可能存在某种形式的意外信息暴露或上下文泄露的疑问。

在目前阶段，仍然没有公开证据证明实际的跨用户私人数据泄露。然而，此案例凸显了与现代 LLM 系统相关的几个重要安全问题：

- 隐藏或保留的特殊标记可能会无意中暴露内部推理机制。
- 不正确的提示解析可能允许用户触发不稳定的模型状态。
- 共享推理基础设施可能会增加请求之间上下文污染的风险。
- 当注入特殊标记时，推理模式模型可能会暴露隐藏的思维链处理行为。
- 前端提示格式化和后端推理管道可能与独立的本地部署不同。

从 LLM 安全的角度来看，此案例也可以被视为一种 **特殊标记攻击** 或 **提示状态操纵攻击**，其中精心制作的标记试图干扰模型的内部推理或控制流。即使不存在实际的数据泄露，不稳定的隐藏状态行为本身仍然可能成为安全风险，因为它可能暴露内部提示、隐藏的推理结构或意外的模型行为。
